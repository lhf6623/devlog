# DevLog 原型设计问答

> 记录 DevLog 从需求到原型的设计过程、关键决策与取舍。

---

## 1. 这个项目到底是做什么的？

**最初定位**：一个待办（Todo）应用。

**实际需求**：用户是前端开发，需要记录后端同事交代的工作事项、需求背景、处理过程和结论。每条记录不是简单的一句话任务，而是包含大量上下文的内容。

**最终定位**：**工作记录 + 任务追踪**工具。既不是传统 Todo，也不是重文档工具，而是介于两者之间的轻量工作日志。

---

## 2. 为什么不做传统待办，而要做成记录型？

传统待办（如 Todoist、Apple Reminders）的字段太简单：
- 标题
- 截止日期
- 优先级
- 完成 checkbox

用户的工作场景是：
- 后端/产品说了什么
- 接口/需求什么时候来
- 实现过程中遇到什么坑
- 最终结论是什么

这些信息需要一块比较大的内容区域来承载，所以核心设计从"任务列表"变成了"记录卡片 + 详情面板"。

---

## 3. 市面上有没有类似产品？参考了谁？

调研了主流产品，大致分为几类：

| 类型 | 代表 | 特点 |
|------|------|------|
| 极简待办 | Apple Reminders、Google Tasks | 字段极少，不适合记内容 |
| 项目管理 | Todoist、Things 3 | 分类清晰，但内容仍是备注字段 |
| 数据库页面 | Notion、Anytype | 内容能力强，但太重 |
| 大纲节点 | Tana、Roam、Workflowy | 任务和笔记一体，学习成本高 |

参考最多的是 **Things 3** 和 **Todoist** 的三栏结构：左侧分类、中间列表、右侧详情。但内容编辑能力比它们更强。

---

## 4. 为什么选择三栏布局？

三栏布局是桌面工具的经典模式：

- **左侧**：筛选维度（时间、状态、类别）
- **中间**：记录列表，快速扫读
- **右侧**：详情面板，读和写内容

这个结构能同时满足：
- 快速定位（左+中）
- 沉浸式编辑（右）
- 内容再长也不影响列表浏览

最终比例：左侧 168px + 中间列表 320px + 右侧详情自适应。

---

## 5. 状态字段为什么不是固定的待办流程，而是四个状态？

最初考虑过简单工作流：待处理 → 进行中 → 已完成。

但用户反馈："本周完成"听起来有成就感，没必要限制。所以最终状态精简为：

- 待处理
- 进行中
- 本周完成
- 已归档

全部由用户手动切换，不做自动流转。这样更随意，符合个人工具的使用习惯。

---

## 6. "本周完成"这个状态是怎么来的？

这是用户自己提出的。传统待办通常用"已完成"，但用户觉得"本周完成"更有成就感，也符合"按周回顾工作"的场景。

这一定位反过来强化了：
- 时间筛选默认是"本周"
- 列表标题显示"本周记录"
- 周五或周日可以方便回顾这周完成了什么

---

## 7. 为什么用"类别"而不是"来源"？

早期方案里用"来源"表示人/部门（比如"后端-老张"）。但用户觉得"来源"这个词太具体，想要更灵活。最终改为"类别"：

- 后端对接
- 产品需求
- 会议记录

用户可以自行添加新类别。类别既作为筛选维度，也作为记录标签。

---

## 8. 内容编辑为什么用 Markdown，而不是富文本编辑器？

用户选择 Markdown 的原因：
- 不要所见即所得（WYSIWYG）
- 编辑和预览分开，减少干扰
- 前端开发对 Markdown 熟悉
- 方便后续迁移和导出

所以设计了两个 Tab：
- **编辑**：写 Markdown 原文
- **预览**：渲染后的结果

---

## 9. 数据存在哪里？为什么选 JSON？

Tauri 应用的数据可以存在：
- 本地 JSON 文件
- SQLite（通过 sql 插件）
- localStorage（不推荐，容量小且易丢失）

最终选择 **本地 JSON 文件**，因为：
- 实现最简单
- 结构清晰，便于后续迁移到 SQLite 或做导入导出
- 用户明确提到"要考虑以后迁移"

JSON 结构示例：

```json
{
  "records": [
    {
      "id": "uuid",
      "title": "订单详情页新增退款入口",
      "content": "...",
      "status": "in_progress",
      "category": "后端对接",
      "createdAt": "2026-06-24T10:00:00",
      "updatedAt": "2026-06-24T16:00:00"
    }
  ],
  "categories": ["后端对接", "产品需求", "会议记录"]
}
```

## 10. 为什么把项目名从 `to-do` 改成 `DevLog`？

原项目名 `to-do` 对应的是"待办"，但产品定位已经是"工作记录"。

候选名：
- 工作志
- 周志
- DevLog
- 事务志
- 跟进簿

用户选了 **DevLog**，因为更贴合开发者身份，国际化也更好。

命名规范：
- `package.json` / `Cargo.toml` 包名：`devlog`
- Tauri 可读名：`DevLog`
- Bundle ID：`com.devlog.devlog`
- Rust lib 名：`devlog_lib`

---

## 11. 视觉风格经历了几次迭代？

**第一阶段**：原生 HTML 控件 + UnoCSS 原子类。界面比较粗糙，按钮和输入框在 Windows 下显得生硬。

**第二阶段**：接入 Naive UI。组件更规整，但风格偏系统默认，左侧菜单出现禁用态的问题。

**第三阶段**：用户发了一张参考图，要求按那个风格重新设计。最终改为：
- 暖白背景 `#f8f7fa`
- 紫色主色 `#6d5dfc`
- 左侧紧凑自定义菜单
- 列表圆角卡片 + 悬停抬升效果
- 详情区简洁紧凑，底部下划线 Tab 切换

---

## 12. 参考图和实际实现的主要差异？

参考图特点：
- 左侧紫色圆点标题
- 小字分组标题
- 筛选项紧凑排列
- 中间列表是圆角卡片，有阴影
- 右侧详情头部信息横向排列
- 编辑/预览是类似胶囊的按钮，但当前态有下划线

实际实现基本还原了这些特征，但做了少量适配：
- 状态用颜色圆点 + 标签
- 保存状态在底部显示
- 保留了显式"保存"按钮，让用户有控制感

---

## 13. 有哪些功能被明确放弃了？

- 所见即所得的 Markdown 编辑器
- 自动归档和自动完成
- 已取消 / 等待反馈 状态
- 优先级字段（红黄蓝标签）
- 多级任务清单
- 多人协作
- 复杂的响应式适配

---

## 14. 搜索功能怎么设计？

搜索框放在列表顶部，按以下字段搜索：
- 标题
- 内容
- 类别

搜索时忽略当前筛选条件，直接显示匹配结果。清空搜索后恢复筛选。

---

## 15. 保存交互是怎样的？

采用自动保存 + 手动保存按钮：
- 标题失焦 600ms 后自动保存
- 内容输入 800ms 防抖后自动保存
- 状态/类别切换后立即保存
- 底部显示"保存中…"和"已保存"状态
- 同时保留显式"保存"按钮，给用户提供确定感

---

## 16. 接下来还可以做什么？

1. 真正的 Tauri 构建验证（完整 `pnpm tauri build`）
2. 导出 / 导入 JSON，方便迁移
3. 快捷键支持（Ctrl+N 新建、Ctrl+S 保存等）
4. 代码块语法高亮
5. 数据量大时从 JSON 切换到 SQLite
6. 记录项的右键菜单或快捷操作
7. 拖拽排序 / 自定义分类排序
